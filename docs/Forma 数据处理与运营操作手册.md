# Forma 数据处理与运营操作手册

> 适用人群：运营人员、数据标注员、交付工程师、项目经理  
> 目的：帮助各类角色**理解 Metax RAG 界面中的处理结果，定位常见问题，并与相关团队协同改进数据质量**。

---

## 1. 工具与背景简介

### 1.1 Forma 是做什么的？

一句话概括：**把“死”文档变成“活”数据**。

- “死”文档：PDF、Word、PPT、图片、扫描件等，人可以看，但系统很难精准理解和搜索。
- “活”数据：结构化的 Markdown 文本、可编辑的问答列表（QA），可以直接被搜索、问答机器人、知识库系统使用。

可以将 Forma 理解为一个“**智能排版员 + 专家助理**”组合：

- 负责将文档拆分为一段一段的内容。
- 尽量保持标题、段落、表格、图片说明等结构信息。
- 在此基础上，将内容提炼成一条条“问题-回答”的知识项（QA）。

### 1.2 运营与交付角色在流程中的位置

在实际项目中，运营、标注和交付人员主要是通过 **Metax RAG 后台界面** 间接使用 Forma：

- 在 Metax RAG 中上传或选择待处理文档；
- 在知识库详情页、文档预览页查看 Forma 生成的 Markdown 和 QA 结果；
- 在上线前对结果进行审核、必要的人工修订和确认；
- 在发现系统性问题时，将“现象 + 初步判断”记录下来，供技术团队分析策略或配置是否需要调整。

本手册的目标是提供一份**围绕 Metax RAG 界面的“现象-诊断-调整方向”参考手册**。

## 2. 核心处理模式（核心概念）

Forma 内部有三种主要处理模式，可以类比为“**三种拍照方式**”：

- `fast`：手机随手拍，清晰度一般，但**快**、省电。
- `deep`：单反+三脚架，构图和细节都很好，但**慢**、费电。
- `auto`：先用手机随手拍，发现糊了，再自动换成单反重拍。

> 在与 Metax RAG 集成的场景下，后台接口当前版本**始终使用 `auto`（智能版）策略**，无需判断具体使用了 Fast 还是 Deep，只需要通过**观察结果特征判断“效果是否达标”“是否需要更深度处理”**，再将这些判断反馈给技术团队即可。

### 2.1 Fast 模式（极速版）

- **概念理解**  
  类似将文档中的文字“复制粘贴”出来，并对图片执行基础的“扫一眼认字”（OCR）。

- **适用场景：**

  - 清晰的电子 PDF、标准 Word 文档、**文字为主**的 PPT。
  - 对图表描述精细度要求不高、以文字信息完整为主的场景。

- **优点：**

  - **处理速度快**：适合大批量文档快速处理。
  - **成本低**：主要使用本地解析和本地 OCR，模型调用成本较低。

- **缺点：**

  - **难以理解复杂图表含义**：通常只提取图表上的文字，无法直接给出“上升趋势、对比关系”等解释。
  - **对扫描件不友好**：如果 PDF 实际为扫描图片，Fast 模式可能只得到极少或几乎没有文字。
  - **表格结构可能受损**：复杂表格在 Markdown 中可能出现列对不齐、合并单元格信息弱化等情况。

**典型结果特征：**

- 文本基本完整，但**图表多为标题，缺少说明性文字**。
- 表格内容存在，但部分列挤在一起或分行不够整齐。
- 扫描版 PDF：可能出现单页只剩几行字，甚至接近空白的情况。

### 2.2 Deep 模式（专家版）

- **概念理解**  
  类似请一位“读图能力较强的专家”通读文档，再**用自然语言进行完整复述**。

- **内部原理：**

  - 系统将页面或 PPT 幻灯片渲染为图片。
  - 图片交由“会看图的 AI”（视觉语言模型 VLM），在图像基础上生成 Markdown 描述。
  - 对 PPT 复杂页，会结合 LibreOffice（专门用于文档渲染的工具）渲染结果和原始文字进行“混合解析”。

- **适用场景：**

  - 扫描件、拍照文档、图片居多的资料。
  - 各种财报、业务报表、架构图、流程图等**图表密集**文档。
  - 对图表含义、趋势解释要求高的项目。

- **优点：**

  - **具备图表理解能力**：可以补充“上升趋势”“两条线对比”“A 大于 B”等描述。
  - **排版还原度较高**：复杂 PPT 幻灯片内容能够较完整地还原。
  - 对扫描件、图片型内容的处理效果明显优于 Fast。

- **缺点：**

  - **处理速度相对较慢**：需要逐页调用视觉模型进行理解。
  - **成本较高**：依赖大模型推理，多页文档的处理费用较为敏感。

**典型结果特征：**

- 图表部分不再只是标题，而是包含：
  - “柱状图显示近三年收入逐年上升……”
  - “左侧是 A 模块，右侧是 B 模块，中间用箭头连接……” 等解释性描述。
- 图片下方通常伴随较详细的文字说明。
- Markdown 整体偏“解释性”，文字数量通常多于原文。

### 2.3 Auto 模式（智能版）

- **概念理解**  
  可以视为一个“先尝试成本较低方案，不足时再启用专家模式”的智能流程。

- **内部逻辑：**

  1. 先用 Fast 跑一遍。
  2. 检查结果：
     - 如果文字数量正常、结构合理 → 认为 Fast 足够，不继续升级。
     - 如果文字极少、乱码较多、明显为扫描件 → 自动改用 Deep 再处理一遍（对整份或者部分页面）。

- **适用场景：**

  - 文档质量和类型不太确定的大批量场景。
  - 希望“多数文件快一点，少数难文件精一点”。

- **优点：**

  - 大部分情况下既能保证整体速度，又能在必要时启用 Deep 补救。
  - 对运营/项目团队而言，只需指定 `auto`（当前默认策略），无需逐个文件人工判断策略。

- **缺点：**

  - 某些“介于中间”的情况（例如半清晰半模糊）可能判断不完全准确，需要结合项目反馈持续微调策略。

**典型结果特征：**

- 同一批文件中，部分结果呈现 Fast 风格（简洁、图表解释较少），部分结果呈现 Deep 风格（解释更为丰富）。
- 对于质量极差的扫描件，仍可能出现“Fast 识别质量较低但未被自动升级”的情况，此时需要结合业务侧验收结果进行诊断和反馈。

> 补充说明：
>
> - 通过 HTTP 接口调用转换服务时（当前版本），系统**默认始终使用 `auto`（智能版）策略**；
> - 命令行工具在不显式指定 `-s` 的情况下（例如使用默认配置），也会走 `auto` 策略。

## 3. 文档结果验收清单

本章节的重点是：**在 Metax RAG 后台中如何判断本次文档处理结果是否达标，以及问题大致属于哪一类**。

验收主要看两块：

- 文档转换结果（Markdown）。
- 知识库结果（QA 问答列表）。

这里先讲 Markdown 的验收。

### 3.1 文本完整性

典型查看位置：Metax RAG 知识库详情页中的「文档预览」或「原文-解析结果」对比视图。

关注点：**是否完整、顺序是否合理、结构是否混乱**。

检查要点：

- **是否有大片空白**：
  - 原文存在整页内容，而 Markdown 中仅保留一行或少量文字 → 可优先怀疑扫描件在 Fast 模式下处理失败。
- **是否有大量乱码**：
  - 文本中出现大量异常符号或断字，例如“法 人 代 表”被拆得很碎。
- **段落顺序是否正确**：
  - 标题和段落整体顺序应与原文保持一致，不应出现明显的“跳段”现象。

如果在预览中发现明显问题，可以从两个维度进行处理：

- **界面层处理（Metax RAG）**：
  - 在文档审核界面中直接补充缺失内容、合并被拆散的段落，保证本次发布的数据可用；
- **技术归因参考（供记录或后续排查）**：
  - 大片空白且原文为扫描件：可能是 Fast 路径未能识别扫描件文本，或 OCR/Deep 能力未启用/异常；
  - 大量乱码或断字：可能与原文编码异常、本地解析链路异常或 OCR 质量不足有关。

### 3.2 表格还原度

典型查看位置：Metax RAG 知识库详情页中的「文档预览」中，尤其是财务报表、清单类表格所在章节。

关注点：**表格的“骨架”和“内容”是否都完整保留**。

检查要点：

- Markdown 表格是否：
  - 在列数和大致结构上与原始表格一致。
  - 重要数值是否对齐在正确的列。
- 对特别复杂的合并单元格：
  - 合并效果可能无法百分百还原，但至少应保证**信息不丢失**，例如部分合并标题被拆成多行，但内容仍然存在。

典型问题表现：

- 多列表格被压成 1-2 列，所有内容挤在一起。
- 表头和数据混在同一列，看不清哪列是“指标”、哪列是“数值”。

如果发现表格结构严重错位，可以结合以下方式处理：

- **界面层处理（Metax RAG）**：
  - 在知识库或文档审核界面中手工调整关键表格（如调整列分隔、补充重要指标），保证当前知识库版本可用；
- **技术归因与调整方向参考**：
  - 多列表格被压成 1-2 列：可能是原始版式复杂、列分隔不易识别，或转换阶段缺乏针对该类表格的专门处理；
  - 表头和数据混在一起：可能需要在转换策略中增强表头识别、或在特定文档类型上启用更保守的 Deep 路径。

### 3.3 图片和图表描述

典型查看位置：Metax RAG 中带有图表或流程图的章节，在预览中关注「图片附近的说明文字」。

关注点：**图片/图表是否被充分“读懂”，而不仅仅是抄录标题**。

检查要点：

- 针对 PPT 或报告中的折线图、柱状图、饼图等：
  - 若只出现图名/标题，缺少“增长/下降/对比关系”说明 → 更接近 Fast 结果。
  - 若包含“趋势/结论/对比”等自然语言描述 → 更接近 Deep 结果。
- 针对架构图、流程图：
  - 若仅保留“系统结构图”等标题 → 倾向于 Fast。
  - 若包含较完整的结构描述，例如“左侧为用户端，中间为网关，右侧为业务服务” → 倾向于 Deep。

在图片和图表描述存在明显缺失时，可以：

- **界面层处理（Metax RAG）**：
  - 为关键图表补充结论性说明，如“近三年收入总体上升”“左侧为用户端，右侧为业务服务”等；
- **技术归因与调整方向参考**：
  - 仅有图名/标题，无趋势或结论：可能是当前页面走了 Fast 路径，仅提取文字而未启用 Deep 分析；
  - 流程图、架构图只有标题：可能需要检查 LibreOffice 是否启用，或在包含大量 SmartArt/图表的 PPT 上优先考虑 Deep 路径和更适配的 Prompt。

## 4. 知识库生成验收（QA 结果）

Forma 会将整理好的知识组织成一份“问答列表”，并在前端以**可编辑的 QA 界面**形式展示，供运营和交付人员逐条检查、修改与确认。

### 4.1 字段说明

常见列有：

- `question`：问题。
  - 例如：“如何去除带字幕的视频中的字幕？”
- `answer`：回答。
  - 例如：“可以使用基于帧差和 OCR 的混合方案……”。
- `category`：分类。
  - 理解为“所在章节的完整路径”，类似“目录层级”。
  - 例如：“素材抽取流水线 > 字幕去除 > 本地实现方案”。

可以将 `category` 理解为：**该条问答来源于原文中的哪一节**。

### 4.2 验收重点 1：问题和回答是否“像人话”

检查点：

- `question`：
  - 是否是用户可能会真实问出口的问题？
  - 是否过于细碎，例如只针对某一行表格数字或单个参数，而业务侧更关注整体使用方法或关键配置？
- `answer`：
  - 是否直接回应了问题本身，没有引入过多无关背景？
  - 是否把关键条件、限制、例外情况说清楚？

在验收记录中，可以简要描述观察到的模式，例如：

- “QA 中问题多集中在单个字段或数值上，对典型使用场景覆盖不足。”

### 4.3 验收重点 2：分类（Category）是否合理

回忆一下：

- QA-V2 不再自己“发明”一套分类体系，而是直接沿用原始文档的**标题层级**。
- `category` 就是把标题链用 `>` 串起来，例如：
  - 一级标题：产品功能介绍
  - 二级标题：计费规则
  - 三级标题：按量计费
  - 最终 category：`产品功能介绍 > 计费规则 > 按量计费`

验收方法：

1. 随机抽取若干条 QA，检查：
   - 问题/答案是否确实与对应章节内容一致；
   - 是否存在明显“跑题”情况，例如关于退款规则的问题被归入“产品功能介绍”章节。
2. 针对某个重要章节：
   - 该章节是否具有足够数量的 QA 覆盖其核心内容；
   - `category` 是否与原始文档层级一一对应。

在记录异常时，可以直接描述发现的情况及影响，例如：

- “《计费与退款规则》章节下包含多条与‘产品安装步骤’相关的 QA，分类与内容不匹配，需检查该章节的切分与 QA 提炼策略。”

## 5. 常见异常与排查参考

本章以 **Metax RAG 界面中可见的“异常现象”** 为入口，给出对应的 **可能技术原因** 与 **调整方向参考**。这些内容既可用于运营侧在日常验收时快速定位问题类型，也可作为后续与技术团队排查时的背景材料。

### 5.1 总体诊断视角

在发现结果异常时，可以从三个角度初步归类：

1. **可见性问题 vs. 理解问题**
   - 可见性问题：看不到内容或大量乱码，多与 OCR、解析链路、扫描件质量有关；
   - 理解问题：内容可见但没有抓住重点，多与是否启用 Deep、Prompt 设计有关。
2. **局部问题 vs. 全局问题**
   - 局部问题：集中在个别页、个别图表或特定章节；
   - 全局问题：整份文档结构或表述风格都不符合预期。
3. **文档质量限制 vs. 系统可优化空间**
   - 原文极度模糊或内容本身混乱时，即使优化策略也难以显著改善；
   - 结构清晰但解析结果明显退化时，更可能属于系统可优化空间。

| **D. 同一批文档中，电子版表现良好，扫描件或图表型文档效果明显偏低**<br>在同一知识库导入任务中，有的文档结构清晰、内容完整，有的几乎不可用。                         | - 使用 Auto 策略时，系统会根据初步结果自动决定是否升级到 Deep；<br>- 对于介于“清晰电子文档”和“严重扫描件”之间的灰度情况，复杂度判断可能不够敏感；<br>- 部分文档类型（例如带少量图片的 PDF）既不像典型扫描件，又不完全是结构化电子文档。 | - 将明显为扫描件/图表型的文档独立为单独任务，使用更激进的 Deep 配置；<br>- 调整 Auto 策略的阈值，使“文字极少/低置信度”更容易触发 Deep；<br>- 在项目侧标注这类文档为“复杂版式”，纳入专项评估。                                            |
| **E. Markdown 基本正确，但 QA 问题粒度不符合业务预期**<br>在 Metax RAG 的 QA 预览中，有时会看到大量过细或过泛的问题，不利于直接部署为 FAQ。                         | - QA-V2 提炼时的粒度与当前项目的知识使用方式不一致（如偏参数级而非场景级，或反之）；<br>- Prompt 中对“面向谁”“用于什么场景”的说明不够明确。                                                                                             | - 根据业务需要，收集目标 QA 示例，作为 QA-V2 Prompt 的参考样本；<br>- 在 Prompt 中明确“问题应更偏向场景/任务级”或“聚焦关键配置点”；<br>- 在 Metax RAG QA 审核界面中对问题进行归并或调优，作为后续策略调优的输入样本。                    |
| **F. 文档较长但 QA 数量明显偏少，看起来“不够多”**<br>在 QA 预览中，原始文档信息量很大，但生成的 QA 条数相对有限。                                                   | - QA-V2 会判断“是否有提问价值”，不会对每一个陈述句都强行生成问题；<br>- 对于主题较复杂或内容较发散的长文档，系统更倾向于生成覆盖面较广的 QA，而不是大量细粒度问答。                                                                     | - 若当前 QA 已覆盖主要业务问题，可以将“QA 数量偏少”视为正常现象，无需追求“越多越好”；<br>- 如需更多 QA，可考虑在 Prompt 中明确期望的问题类型和粒度，或对原始文档做更细颗粒度的章节切分，使每个部分更聚焦单一主题。                       |

> 说明：表中“调整方向参考”不是固定操作指令，而是供项目团队在结合自身环境、资源和优先级时做决策的技术参考。

## 6. 小结

在 Metax RAG 的实际使用中，Forma 作为后台文档处理引擎，负责将多源异构文档统一转换为结构化 Markdown 和 QA；运营、标注和交付人员则通过 Metax RAG 界面完成审核、修订和上线决策。

本手册给出的检查要点和异常场景，有助于在日常工作中更快回答三个问题：

- 当前这批文档/QA 是否适合直接进入知识库？
- 如果不适合，大致属于哪一类技术或文档质量问题？
- 需要在 Metax RAG 界面做哪些调整，是否有必要进一步评估策略或配置？

在这一基础上，团队可以逐步形成稳定、可复用的“人机协作”流程：让 Forma 负责大规模、标准化的文档转换与知识提炼，让人工审核聚焦于关键判断和策略调整，从而持续提升整体知识库质量和维护效率。
