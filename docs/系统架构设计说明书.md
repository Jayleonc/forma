# Forma 智能文档处理引擎

## 系统架构设计说明书

> 适用对象：技术人员  
> 文档范围：整体架构设计、关键子系统与数据流、非功能性设计（性能与扩展性）

---

## 1. 产品背景与定位

### 1.1 业务背景

在企业知识管理和智能客服场景中，海量的 PDF、DOCX、PPTX、图片等非结构化文档难以直接被搜索、分析和问答系统消费。传统做法通常依赖人工整理或简单 OCR，存在：

- **结构缺失**：文档层级、标题体系、表格与图表语义丢失。
- **内容割裂**：同一知识点分散在不同章节、不同文件，难以统一抽取。
- **处理成本高**：复杂版式、扫描件、图片型 PPTX 需要大量人工介入。

### 1.2 Forma 的定位

Forma 将自身定位为 **非结构化文档的智能 ETL 引擎**，面向下游搜索 / 检索增强生成 (RAG) / FAQ 知识库系统，完成：

- **Extract（抽取）**：对 PDF、DOCX、PPTX、图片等进行文本、结构、视觉内容抽取。
- **Transform（转换）**：统一转化为结构化 Markdown，并进一步提炼为问答对（QA）。
- **Load（加载）**：以 CSV / JSONL 等结构化格式输出，便于加载至搜索引擎或知识库。

在整体企业架构中，Forma 通常处于：

> 上游：文档管理系统、文件存储（对象存储、DMS）。  
> 中游：作为独立的 **文档智能处理服务**（CLI/HTTP API）。  
> 下游：搜索/RAG 服务、FAQ 机器人、知识图谱系统等。

在具体落地场景中，Forma 既保持作为通用文档 ETL 引擎的中立性，又针对 **RAG 型知识库系统** 做了深度优化。其中，Metax RAG 是 Forma 的典型下游消费方：

- Metax RAG 聚焦于 **知识库管理、向量检索与 Agent 对话编排**，不直接处理原始文件解析与清洗。
- Forma 则承担“脏活累活”：负责多格式文档的解析、结构化转换、OCR/VLM 深度理解以及 QA-V2 问答知识提炼。
- 两者形成清晰的上下游分工：Forma 输出高质量 Markdown 与 QA 数据，Metax RAG 负责后续入库、向量化及在线检索问答。

### 1.3 在 Metax RAG 生态中的系统集成

在 Metax RAG 体系中，Forma 作为 **核心上游数据处理引擎**，通过 HTTP 异步接口向 Metax RAG 提供文档转换与知识提炼能力。整体数据流向可以概括为：

1. **文件上传与任务创建（Metax RAG）**

   - 终端用户或上游业务系统将原始文件上传至 Metax RAG（通常经由其 API Gateway 或管理控制台）。
   - Metax RAG 接收文件元信息与存储位置，生成内部任务，并以调用方身份请求 Forma 的转换/QA 接口。

2. **调用 Forma 执行转换与提炼（Forma 作为执行方）**

   - Metax RAG 向 Forma 发起 HTTP 请求，传入：
     - 源文件地址（如对象存储 URL）；
     - 回调地址 `callback_url`；
     - 任务标识 `task_id` 或业务侧 `request_id`（用于关联与幂等）。
   - Forma 返回受理结果（包含 `task_id` 与初始状态），并在内部调度转换或 QA-V2 流水线。

3. **异步回调结果（Forma → Metax RAG）**

   - 任务完成或失败后，Forma 通过 **HTTP 异步回调** 将结果推送至 Metax RAG 预先约定的回调端点：
     - 成功：返回结构化 Markdown 内容、生成的 QA 列表或相应的结果文件地址；
     - 失败：返回错误码与可读的错误信息，便于 Metax RAG 做重试或告警。
   - Metax RAG 作为调度方，通过 `task_id`/`request_id` 将回调结果与原始任务绑定，实现任务生命周期管理与观测。

4. **入库、向量化与在线服务（Metax RAG 作为下游消费方）**

   - Metax RAG 接收 Forma 的 Markdown 与 QA 结果后：
     - 将 Markdown 按业务策略切分为片段，执行向量化并写入向量库；
     - 将 QA-V2 生成的问答对纳入知识库，作为结构化 FAQ 或多轮对话意图的候选知识源；
     - 结合自身的索引、检索与 Agent 机制，对外提供 RAG 检索问答能力。

从系统视角看，二者之间的整体调用链可以抽象为：

> 用户 / 上游业务系统  
> → Metax RAG（API Gateway / 控制面）  
> → [发起转换/QA 任务调用] → Forma（Conversion & QA Engine）  
> → [HTTP 异步回调] → Metax RAG（知识库与向量存储）  
> → [在线检索与推理] → 最终用户。

## 2. 系统总体架构

### 2.1 分层架构概览

从平台视角，Forma 采用四层分层架构：

1. **接入层（Interface Layer）**

   - 提供 CLI (`forma convert` / `forma qa-v2`) 与 HTTP API（FastAPI）。
   - 支持同步 CLI 调用与异步 API 调用（含异步回调）。

2. **调度层（Orchestration Layer）**

   - 负责任务编排和策略路由：根据文件类型与策略（Auto/Fast/Deep）选择合适的处理路径。
   - 管理并发 worker（`CONVERSION_WORKERS` / `QA_WORKERS`），执行异步任务调度与超时控制。
   - 统一处理任务状态（queued / processing / completed / failed）。

3. **引擎层（Engine Layer）**

   - **文档转换引擎（Conversion）**：PDF / DOCX / PPTX / Image 等格式的解析、OCR、VLM 调用。
   - **知识提炼引擎（QA）**：基于层级化上下文注入的 QA-V2 管线，将 Markdown 转为高质量问答对。
   - **视觉引擎（Vision）**：封装视觉语言模型 (VLM) 的调用与结果解析。
   - **OCR 引擎（OCR）**：封装 PaddleOCR 等本地 OCR 能力。

4. **基础组件层（Foundation Layer）**
   - 配置中心（基于 Pydantic 的配置加载与环境变量管理）。
   - 提示词管理（`prompts.yaml`，实现 Prompt 与代码解耦）。
   - 通用工具库（日志、文件系统、错误处理等）。

#### 2.1.1 架构说明

- **接入层**：
  - CLI 客户端 / HTTP 客户端 → 调用 `forma` 命令或 FastAPI 端点。
- **调度层**：
  - 接口控制器接收请求 → 生成 `task_id` → 将任务投递到 **转换任务队列** 或 **QA 任务队列** → 后台 worker 进程异步消费。
- **引擎层**：
  - 转换 worker 根据文件类型和策略路由到 **PDF/DOCX/PPTX/Image 处理器**。
  - QA worker 根据任务类型路由到 QA-V2 **层级化提炼管线**。
  - OCR/Vision 作为子组件被各处理器按需调用。
- **基础组件层**：
  - 所有上层模块通过共享配置、日志与 Prompt 管理器协同工作。

### 2.2 模块化设计思想

Forma 核心代码按业务域拆分为独立模块，形成清晰的“引擎即组件”结构：

- **Conversion 模块**

  - 职责：文档到 Markdown 的转换流水线。
  - 特点：按文件类型（PDF/DOCX/PPTX/Image）划分 `processor`，支持策略路由（Fast/Deep/Auto）。

- **OCR 模块**

  - 职责：统一封装 OCR 引擎（本地 GPU/CPU），对上层暴露简单的文本检测接口。
  - 特点：支持 OCR 降级处理——在部分场景下，优先使用直接文本提取，失败时回退到 OCR。

- **Vision 模块**

  - 职责：与视觉语言模型 (VLM) 交互，接收图片/页面截图，输出结构化 Markdown 描述。
  - 特点：完全通过 `prompts.yaml` 驱动，实现模型无感知的业务升级。

- **QA 模块**

  - 职责：从 Markdown 文档提炼结构化问答知识库（QA-V2）。
  - 特点：采用 **层级化提炼 + 摘要回填** 的单阶段架构，舍弃 V1 的向量化聚类，提升性能与可控性。

- **Shared 模块**
  - 职责：配置管理、Prompt 管理、共用工具函数。
  - 特点：保证上下文配置与提示词在 Conversion / OCR / QA / Vision 中一致使用。

## 3. 关键子系统设计

### 3.1 文档转换引擎

文档转换引擎以 `conversion/workflow` 为核心，围绕不同文件类型提供专业化 `processor`，并基于 **Auto/Fast/Deep 策略路由** 提供统一体验。

#### 3.1.1 策略路由（Auto/Fast/Deep）

- **Fast 策略**

  - 目标：在保证可用性的前提下最大化吞吐，优先选择本地解析和 OCR。
  - 典型路径：
    - PDF：本地文本提取 + 针对图片区域的按需 OCR。
    - DOCX：HTML 转 Markdown 的优先路径，必要时回退到逐段解析。
    - PPTX：优先走“快速路径”（详见后文 PPTX 混合管线）。
    - 图片：本地 OCR 直接提取文本。

- **Deep 策略**

  - 目标：最大化内容保真度和语义质量，以 VLM 为核心，对复杂版式、图表、扫描件进行深度理解。
  - 典型路径：
    - 将页面或幻灯片渲染为图片，调用 Vision 模块。
    - 使用针对场景的 Prompt（如技术架构图分析、财报表格分析）引导 VLM 生成结构化 Markdown。

- **Auto 策略**
  - 目标：在 **体验层面隐藏复杂度**，自动为每个文件选择合适的 Fast/Deep 组合。
  - 逻辑（以源码为准）：
    - 第一步：各文件类型先按各自的 Fast 路径完成一次处理（例如 PDF 本地解析、PPTX 快速路径等）。
    - 第二步：根据处理结果中的 **字符数阈值** 和处理器返回的 **低置信度标记** 做质量评估：
      - 若 `result.text_char_count < THRESHOLD`（`THRESHOLD` 来自 VLM 配置 `AUTO_THRESHOLD`，默认约为 200 字符），或 `result.low_confidence == True`，认为 Fast 结果质量不足。
    - 第三步：在上述条件触发时，进入 **自动升级** 分支，对文件走一次基于 VLM 的 Deep 路径（通过 `VlmParser.parse()` 重新解析原文件）。
  - 对调用方而言，只需选择 `auto`，即可在“足够快”与“必要时升级 Deep”之间取得平衡，无需感知内部阈值细节。

#### 3.1.2 PDF 特殊处理逻辑

- **文本型 PDF**：

  - 使用 `pymupdf` 等库直接解析文字与基本版式信息。
  - 对嵌入图片进行按需 OCR，作为补充内容合入 Markdown。

- **扫描型 PDF**：
  - Fast 路径检测到文本极少时，触发 **OCR 降级处理**：将页面视为图片整体送入 OCR 或 VLM。
  - 在 Auto 策略下，系统基于上述检测自动切换到 Deep 路径。

#### 3.1.3 DOCX 混合处理策略

DOCX 采用两级回退机制，以最大限度保持结构信息：

1. **首选：Mammoth → HTML → Markdown**

   - 优点：对表格、列表、粗体、标题等保留较好，适合内容完整迁移。
   - 用途：作为主路径处理绝大多数标准 DOCX 文档。

2. **回退：python-docx 逐段解析**
   - 当 Mammoth 失败或遇到异常结构时，自动回退。
   - 通过遍历段落与表格节点，保证内容不丢失，即使版式简化。

该设计实现了一种 **结构优先 + 内容兜底** 的处理逻辑。

#### 3.1.4 PPTX 混合管线设计

PPTX 处理是 Forma 在转换引擎中的核心创新，采用 **按页混合管线**：

- **目标**：同时兼顾 **内容保真度**（复杂幻灯片不丢信息）与 **性能**（复杂文件可快速处理）。

##### a) 幻灯片复杂度检测（快速/深度路径选择）

每张幻灯片在进入具体处理逻辑前，会进行复杂度检测：

- **复杂判定规则**：

  - 含有图表 (Chart) / SmartArt / 表格 (Table) → 直接标记为复杂页。
  - 文本字符数低于阈值（如 25 字符） → 视为图片/图示为主 → 标记为复杂页。

- **结果**：将整个 PPT 拆分为两类页：
  - 简单页（走 Fast Path）。
  - 复杂页（走 Deep Path）。

##### b) 快速路径（简单幻灯片）

适用于文本主导的幻灯片：

- 使用 `python-pptx` 提取所有形状中的文字。
- 提取嵌入图片并调用 OCR 引擎识别图片中的文字，将结果附加到幻灯片对应的 Markdown。
- 不依赖外部进程（如 LibreOffice），速度极快，适合多数标准演示文稿。

##### c) 深度路径（复杂幻灯片）

复杂幻灯片采用 **LibreOffice 预渲染 + VLM 视觉分析 + 文本合并** 的混合深度路径：

1. **优化 1：LibreOffice 一次转换，多次复用**

   - 当任意一页被识别为复杂页时：
     - 系统不会为每一页单独调用 LibreOffice。
     - 而是将整个 PPTX **一次性转换** 为一个中间 PDF 文件。
   - 在处理每一张复杂页时：
     - 直接从该 PDF 中按页渲染所需的单页为高分辨率图片。
     - 避免 N 次 libreoffice 进程启动与关闭，显著降低 I/O 与 CPU/内存抖动。

2. **优化 2：混合解析（VLM + 原始文本合并）**

   - 渲染出的幻灯片图片：
     - 送入视觉语言模型 (VLM)，通过专用 Prompt 让模型“理解”图表、布局和视觉元素，输出 Markdown。
   - 为避免 VLM 遗漏文字：
     - 系统并行用 `python-pptx` 提取该页所有文本。
     - 将 VLM 结果与原始文本进行合并，执行去重和排序。
   - 最终输出：
     - 既包含 VLM 的视觉理解结果（对图表和布局的说明），也包含所有原始可见文本，实现 **视觉语义 + 文本完整** 的最大化保真。

3. **降级与兼容**
   - 若环境未安装 LibreOffice：
     - 系统自动跳过复杂页深度解析，仅输出提示信息，不影响整体文件处理完成。
   - 对调用方来说，行为为 **渐进增强**，不会因环境差异导致服务失败。

### 3.2 知识提炼引擎（QA-V2）

QA-V2 是 Forma 从“文档 → 问答知识库”的核心子系统，相比 V1 的“Embedding + 聚类 + 二次合成”架构，V2 采用 **自顶向下的层级化提炼 + 摘要回填 + 标题链分类**，带来更好的性能与稳定性。

#### 3.2.1 QA-V1 回顾：聚类式架构

在代码层面，QA-V1 仍以完整流水线形式存在，作为“全局聚类+合成”的老方案，主要用于早期项目迁移和与历史数据对齐的场景；当前推荐默认使用 QA-V2，新接入项目仅在有强需求时才需要考虑启用 V1。理解 V1 的设计有助于在需要时评估“是否需要回到三阶段聚类式流程”。

V1 的架构特点：

- **阶段 1：原始问答生成**

  - 将 Markdown 切分为内容块，对每块调用 LLM 生成初始 QA 对。

- **阶段 2：全局分类体系生成**

  - 收集所有问题文本。
  - 使用 Embedding（如 Sentence-Transformers）生成向量。
  - 使用 DBSCAN 等聚类算法发现语义簇，并通过 LLM 生成全局主题分类。

- **阶段 3：合成与指派**
  - 对每个簇调用 LLM 进行总结与合成，输出最终标准化 QA 对。

该方案的不足：

- 需要大规模向量化和聚类计算，**性能与资源成本高**。
- 聚类结果与主题命名依赖 LLM，自定义性较强但**结构不稳定**，不利于与原始文档结构对齐。
- 三阶段流水线增加系统复杂度和故障点。

#### 3.2.2 QA-V2：层级化提炼 + 摘要回填

QA-V2 进行了架构收敛与重点强化，整体变为 **单阶段 Top-Down 管线**：

1. **层级化切分与结构建模**

   - 使用层级 Chunker 将 Markdown 按标题层级切分，构建父子关系与 `header_chain`（如 `一级标题 > 二级标题 > 三级标题`）。

2. **自顶向下（Top-Down）层级提炼**

   - 从文档根节点（最高层标题）开始，按层（BFS）向下遍历。
   - 同一层级的兄弟节点可并行处理，提高吞吐。
   - 对每个节点：调用 LLM 对当前文本进行总结与 QA 提炼。

3. **摘要回填（Summary Backfill）+ 层级上下文注入**

   - 子节点提炼时，将其父节点摘要作为背景知识注入 Prompt，实现真正的“层级上下文注入”：
     - 父级摘要提供全局意图和上位语境。
     - 子级提炼在更稳定且聚焦的上下文中进行，减少歧义与冗余。
   - 当不存在父节点（根节点）时，使用轻量的默认父级摘要以稳定模型行为。

4. **标题链分类与直接产出**
   - V2 不再进行全局 Embedding/聚类，也不再生成额外主题层次。
   - 每条 QA 对的 `category` 直接取自其来源块的 `header_chain`，并用 `" > "` 连接：
     - 例如：`素材抽取流水线 > 字幕去除 > 本地实现方案`。
   - 输出为扁平化 QA 列表：`question`、`answer`、`category`，可直接导出为 CSV / JSONL 供下游消费。

#### 3.2.3 架构演进的动机（Why V2）

- **性能与成本**

  - 移除 Sentence-Transformers 和 DBSCAN 等向量化与聚类组件后，流水线大幅简化，减少 CPU/GPU 资源占用和整体延迟。
  - 单阶段 Top-Down 流水线更容易进行水平扩展和并行化控制。

- **结构对齐与稳定性**

  - V1 动态生成的主题分类难以保证与源文档结构、业务定义一致。
  - V2 采用 `header_chain` 作为分类，天然对齐原始文档的章节体系，更适合与业务域模型绑定（如“产品功能 > 计费规则 > 退费策略”）。

- **工程复杂度与可观测性**
  - V2 不再有“二次合成”阶段，所有 QA 对均可直接溯源到具体块与标题链。
  - 故障排查与质量评估更直接，便于接入外部监控与审核流程。

#### 3.2.4 QA-V2 处理流（自顶向下 Top-Down）

以一次 `qa-v2` 调用为例，其高层逻辑如下：

1. **输入**：Markdown 文档。
2. **层级切分**：构建块树，每个节点包含文本与 `metadata.header_chain`。
3. **顶层提炼**：对根节点及一级标题进行摘要与 QA 提炼。
4. **摘要回填**：将父节点摘要注入子节点 Prompt，继续向下提炼。
5. **语言自适应与稳健性**：
   - 根据文本语言分布给出语言偏好提示，优先中文。
   - 极短文本块跳过 LLM 调用，仅生成最小占位摘要，避免无效消耗。
6. **扁平化输出**：
   - 收集所有节点的 `qa_pairs`，统一输出为 CSV/JSONL。
   - 每条 QA 均带有完整 `header_chain`，便于下游查询和过滤。

## 4. 数据流与交互设计

### 4.1 文档转换任务全生命周期

以 HTTP API `POST /api/v1/convert` 为例：

1. **任务创建**

   - 客户系统调用转换接口，传入：`request_id`、`source_url`、`callback_url`。
   - Forma 立即返回 `task_id` 与初始状态 `processing`，请求方无需阻塞等待。

2. **任务调度与下载**

   - 后台 worker 拉取任务：
     - 下载源文件到本地数据目录 (`FORMA_DATA_DIR`)。
     - 根据文件后缀或内容类型解析出文档类型（PDF/DOCX/PPTX/Image）。

3. **策略路由与引擎执行**

   - 在 **API 场景** 下：`/api/v1/convert` 当前实现固定使用 `Strategy.AUTO`，由服务端在内部根据 `text_char_count` 与 `low_confidence` 判断是否升级到 Deep；调用方无需、也不能在请求体中显式指定策略。
   - 在 **CLI 场景** 下：`forma convert` 支持调用方显式指定 `auto` / `fast` / `deep`。
   - 触发相应的文件处理器：
     - 普通 PDF → Fast 文本提取 + OCR；在 AUTO 下，当处理结果字符数过少或被标记为低置信度时，会回落到 VLM 解析。
     - PPTX → 幻灯片复杂度检测 → 快速/深度混合管线。
     - 其他文件类型 → 按各自 Processor 的 Fast 路径输出；在 AUTO 下同样基于 `text_char_count` 与 `low_confidence` 决定是否触发 Deep 路径。

4. **结果生成与存储**

   - 转换结果为 Markdown 文本，按需要写入文件或保存在临时存储。
   - 如后续还需进入 QA-V2，可直接将 Markdown 作为下一阶段任务输入。

5. **异步回调（Asynchronous Callback）**

   - 任务完成或失败后，Forma 调用 `callback_url`，携带：
     - `request_id`（透传用于幂等和关联）。
     - `status`（completed / failed）。
     - 成功：`markdown_content`。
     - 失败：`error_message`。
   - 客户系统可据此进行下一步处理或告警。

6. **超时与错误处理**
   - 通过 `FORMA_CONVERSION_TIMEOUT` 控制单任务最长执行时间。
   - 发生异常时，系统确保：
     - 日志中记录完整的错误栈。
     - 回调中包含可读的错误说明，不影响整体服务可用性。

### 4.2 QA 生成任务全生命周期

以 `POST /api/v1/generate-qa` 为例：

1. **任务创建**：

   - 客户系统传入 `markdown_content` 与 `callback_url`，立即获得 `task_id` 和 `processing` 状态。

2. **调度执行**：

   - QA worker 拉取任务，调用 QA-V2 管线执行 Top-Down 层级提炼与 QA 生成。

3. **结果回调**：

   - 成功：通过 `faq_json` 或已持久化的 CSV 路径返回。
   - 失败：通过 `error_message` 告知具体失败原因。

4. **并发与隔离**：
   - 由 `QA_WORKERS` 决定并发 worker 数量，确保与转换任务相互隔离，互不影响性能指标。

### 4.3 Metax RAG 调用接口规范示例

本节从“系统集成协议”的角度，给出 Metax RAG 作为调用方访问 Forma HTTP API 的字段规范示例。

#### 4.3.1 `POST /api/v1/convert`（文档转换）

**请求 JSON 示意**：

```json
{
  "request_id": "rag-20241001-0001",
  "source_url": "https://oss.example.com/bucket/path/to/file.pdf",
  "callback_url": "https://metax-rag.example.com/api/forma/callback/convert",
  "extra": {
    "biz_type": "knowledge_base_import",
    "tenant_id": "tenant-001"
  }
}
```

**字段说明**：

- `request_id`：
  - 由 Metax RAG 生成的业务请求标识，用于幂等与日志追踪；
  - Forma 原样透传到回调结果中，不参与业务语义判断。
- `source_url`：
  - 待转换文档的可访问地址，一般为对象存储 URL；
  - Forma 作为执行方仅负责按该地址下载文件，不对权限策略做额外假设。
- `callback_url`：
  - Metax RAG 提供的异步回调端点；
  - Forma 在任务完成或失败后向该地址发起 HTTP 请求推送结果。
- `extra`：
  - 预留的业务扩展字段，由 Metax RAG 自定义；
  - Forma 不解析其内部结构，但会在回调结果中一并透传，方便 Metax RAG 做多租户或业务分流。

**同步响应示意（Forma → Metax RAG）**：

```json
{
  "task_id": "conv-8f3e2c9a",
  "status": "processing",
  "request_id": "rag-20241001-0001"
}
```

- `task_id`：Forma 内部任务唯一标识，用于后续观测与排错；
- `status`：当前任务状态，初始为 `processing`；
- `request_id`：原样回显调用方提供的业务请求标识。

**异步回调示例（Forma → Metax RAG `callback_url`）**：

成功场景：

```json
{
  "task_id": "conv-8f3e2c9a",
  "request_id": "rag-20241001-0001",
  "status": "completed",
  "markdown_content": "# 文档标题\n...",
  "extra": {
    "biz_type": "knowledge_base_import",
    "tenant_id": "tenant-001"
  }
}
```

失败场景：

```json
{
  "task_id": "conv-8f3e2c9a",
  "request_id": "rag-20241001-0001",
  "status": "failed",
  "error_message": "download timeout from source_url",
  "extra": {
    "biz_type": "knowledge_base_import",
    "tenant_id": "tenant-001"
  }
}
```

其中：

- `markdown_content`：
  - Forma 生成的结构化 Markdown 文本，Metax RAG 后续可直接用于切分与向量化；
  - 对于超大文档或需要文件化落盘的场景，可替换为 `markdown_url` 等形式。
- `error_message`：
  - 人类可读的错误说明，便于 Metax RAG 做重试/告警；
  - 不承诺稳定的机器可解析枚举值，若需则可在此基础上扩展错误码字段。

#### 4.3.2 `POST /api/v1/generate-qa`（QA-V2 知识提炼）

**请求 JSON 示意**：

```json
{
  "request_id": "rag-20241001-0002",
  "markdown_content": "# 文档标题\n...",
  "callback_url": "https://metax-rag.example.com/api/forma/callback/qa",
  "prompt_profile": "default_zh",
  "extra": {
    "kb_id": "kb-metax-product",
    "tenant_id": "tenant-001"
  }
}
```

**字段说明**：

- `request_id`：
  - 由 Metax RAG 生成的业务请求标识，与转换接口含义一致；
- `markdown_content`：
  - QA-V2 的输入 Markdown，一般直接来源于上一阶段转换结果；
  - 若 Markdown 过大，也可通过内部约定扩展为文件地址字段；
- `callback_url`：
  - QA 结果的异步回调端点；
- `prompt_profile`：
  - 可选字段，用于选择不同的 Prompt 配置（例如“产品手册场景”、“运维手册场景”）；
  - 若省略则使用服务端默认配置；
- `extra`：
  - 承载知识库 ID、多租户 ID 等业务信息，Forma 透传不解析。

**同步响应示意（Forma → Metax RAG）**：

```json
{
  "task_id": "qa-57ab1390",
  "status": "processing",
  "request_id": "rag-20241001-0002"
}
```

**异步回调示例（Forma → Metax RAG `callback_url`）**：

成功场景：

```json
{
  "task_id": "qa-57ab1390",
  "request_id": "rag-20241001-0002",
  "status": "completed",
  "faq_json": [
    {
      "question": "什么是 Forma?",
      "answer": "Forma 是一个面向企业级文档智能处理的后端引擎...",
      "category": "产品概述 > 定位"
    }
  ],
  "extra": {
    "kb_id": "kb-metax-product",
    "tenant_id": "tenant-001"
  }
}
```

失败场景：

```json
{
  "task_id": "qa-57ab1390",
  "request_id": "rag-20241001-0002",
  "status": "failed",
  "error_message": "llm provider timeout",
  "extra": {
    "kb_id": "kb-metax-product",
    "tenant_id": "tenant-001"
  }
}
```

对 Metax RAG 来说：

- `faq_json` 可直接写入自身知识库或转存为 CSV/JSONL；
- 结合 `category` 字段（来源于 QA-V2 的 `header_chain`），可以按章节体系对知识进行分组与检索；
- 通过 `request_id` 与 `extra` 中的 `kb_id`/`tenant_id` 等字段，可将 Forma 的异步输出精确路由到对应的租户与知识库空间。

## 5. 非功能性设计

### 5.1 可扩展性

- **Prompt 分离与策略可配置**

  - 所有与 VLM/LLM 交互的 Prompt 全部集中定义在 `prompts.yaml`：
    - 支持根据业务场景自定义不同 Prompt（如“财务报表分析”、“技术架构图解读”）。
    - 调用方通过 CLI 参数或 API 字段指定 Prompt 名称，无需修改代码。
  - 这种设计实现了 **逻辑代码与语义策略的解耦**，便于快速迭代与 A/B 测试。

- **格式与引擎的横向扩展**

  - Conversion 通过 `processors` 目录按文件类型拆分，新格式（如 XLSX）只需新增 Processor 与策略路由即可。
  - OCR 与 Vision 引擎通过统一接口封装，可在不影响上层的前提下替换模型或厂商。

- **部署与容器化**
  - 提供 Docker 镜像构建与运行脚本，便于在客户环境中快速部署。
  - 通过环境变量配置 worker 数量与时间限制，支持弹性伸缩。

### 5.2 性能与资源利用

- **GPU 加速与自动依赖选择**

  - 安装脚本会根据操作系统选择合适的 Paddle/torch 版本：
    - Linux 环境下自动安装 GPU 版本，充分利用 GPU 加速 OCR 与可能的模型推理。
    - macOS/Windows 环境默认使用 CPU 版本，保证可用性。
  - 这一机制使 Forma 可以 **在不改动代码的情况下** 自动适配不同硬件资源。

- **LibreOffice 进程复用（PPTX 预渲染优化）**

  - 对于包含大量复杂幻灯片的 PPTX，若对每一页单独调用 LibreOffice 转 PDF，系统将陷入严重的 I/O 和进程开销。
  - Forma 采用“一次转换，多次复用”策略：
    - 仅在检测到复杂页的前提下一次性将 PPTX 转为 PDF。
    - 随后按需从该 PDF 中渲染具体页面，配合 VLM 完成深度解析。
  - 实测上，随复杂页数量增加，该策略能显著降低整体处理时间和资源消耗。

- **并发 Worker 管理**

  - 通过 `CONVERSION_WORKERS` 与 `QA_WORKERS` 控制不同任务类型的并行度。
  - 提供独立的 worker 池，使重型的 Deep 转换任务不会阻塞 QA-V2 或轻量任务。
  - 配合 FastAPI 异步模型与异步回调机制，可在高并发场景下保持稳定响应。

- **BFS 分层并行（QA-V2）**
  - QA-V2 利用 BFS 按层处理节点，在保证父摘要已就绪的前提下，最大化同层兄弟节点并行数。
  - 这种设计既保留了 Top-Down 的语义一致性，又充分利用多核资源，适合长文档与多章节场景。

---

## 6. 总结

Forma 作为一个面向企业级文档智能处理的后端引擎，通过：

- 在转换侧采用 **Auto/Fast/Deep 策略路由** 与 **PPTX 混合管线 + LibreOffice 预渲染优化**，兼顾复杂文档的保真度与整体吞吐。
- 在知识提炼侧用 **QA-V2 的层级化提炼 + 摘要回填 + 标题链分类** 取代传统 Embedding 聚类方案，降低复杂度与成本的同时提升稳定性与可解释性。
- 通过 Prompt 分离、模块化引擎设计与异步回调机制，确保系统在扩展性、性能和对接体验上的平衡。

从生态角度看，该架构既适合作为独立服务部署于客户私有环境，也可以通过 HTTP 异步接口与 Metax RAG 等 RAG 型知识库系统集成：Forma 作为上游文档智能处理引擎，负责文档解析、结构化转换与 QA 提炼；Metax RAG 作为下游知识库与向量检索平台，负责数据入库、向量化与在线检索问答，共同构成非结构化文档到知识服务的一体化链路。
